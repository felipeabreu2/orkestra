# Orkestra — Fase 13 (UI/UX polish) — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Elevar a aparência e o acabamento do app de "design mínimo funcional" para uma estética **premium coesa de ferramenta de dev** (dark, tipografia limpa, espaçamento generoso, motion sutil), e resolver os ajustes de UX acumulados das fases anteriores.

**Estética (direção — AJUSTÁVEL pelo usuário no checkpoint):** dark premium no espírito de ferramentas de desenvolvedor modernas (fundo quase-preto azulado, texto off-white, **um** acento — violeta/índigo — usado com parcimônia, cantos suaves, sombras de elevação sutis, transições de 150–220ms ease-out). Sem nomear produtos. O usuário tem interesse forte em premium/motion — esta direção é um ponto de partida sensato que ele pode redirecionar (paleta, densidade, vibe).

**Architecture:** Um arquivo de **design tokens** (CSS custom properties em `:root`) centraliza cores, tipografia, espaçamento, raio, sombra e durações de motion. Os componentes (que hoje usam inline styles) passam a referenciar `var(--token)` — mudança incremental, sem reescrever a árvore. Uma folha `base.css` define fonte, fundo e reset fino. Nenhuma mudança de lógica/estado; nenhum teste de comportamento deve quebrar.

**Tech Stack:** CSS puro (custom properties), sem libs de UI novas. Vitest continua verde (mudanças são visuais).

## Global Constraints

- **Zero regressão funcional:** nenhuma mudança de lógica, IPC, store, ou segurança. Todos os 186 testes continuam verdes; `typecheck`/`build` limpos.
- Renderer não ganha imports proibidos. Nomenclatura sem marcas do Maestri.
- Acessibilidade básica: foco visível, `role`/`aria` onde falta (palette/modais), contraste adequado.
- Motion respeita `prefers-reduced-motion` (desabilita transições se o usuário preferir).

### Task 1: Design tokens + consolidação de UX (os Minors deferidos)

**Interfaces:** tokens CSS globais; correções de UX pontuais. Sem novas exports de lógica.

- [ ] **Step 1: `tokens.css` + `base.css`**

`tokens.css` — `:root` com custom properties (valores iniciais da direção premium; ajustáveis):
```css
:root {
  /* superfícies */
  --bg-0: #0b0d12;      /* fundo do canvas */
  --bg-1: #12151c;      /* painéis/nós */
  --bg-2: #1a1e27;      /* elevação/hover */
  --border: #262b36;
  --border-strong: #333a48;
  /* texto */
  --text-1: #e6e9ef;    /* primário */
  --text-2: #a2abbd;    /* secundário */
  --text-3: #6b7488;    /* terciário/placeholder */
  /* acento */
  --accent: #7c6cff;    /* violeta/índigo */
  --accent-weak: #7c6cff22;
  /* estados */
  --ok: #3fb984; --warn: #e0a13a; --err: #e5615f;
  /* forma */
  --radius-sm: 6px; --radius: 10px; --radius-lg: 14px;
  --shadow-1: 0 1px 2px #0006, 0 2px 8px #0004;
  --shadow-2: 0 8px 30px #0008;
  /* tipografia */
  --font-ui: -apple-system, BlinkMacSystemFont, 'Segoe UI', Inter, system-ui, sans-serif;
  --font-mono: 'SF Mono', 'JetBrains Mono', Menlo, Consolas, monospace;
  /* motion */
  --dur-1: 120ms; --dur-2: 200ms; --ease: cubic-bezier(0.2, 0.6, 0.2, 1);
}
@media (prefers-reduced-motion: reduce) { :root { --dur-1: 0ms; --dur-2: 0ms; } }
```
`base.css` — `html,body,#root { height:100%; margin:0 }`, `body { background: var(--bg-0); color: var(--text-1); font-family: var(--font-ui); font-size: 13px; -webkit-font-smoothing: antialiased }`, `* { box-sizing: border-box }`, foco visível global `:focus-visible { outline: 2px solid var(--accent); outline-offset: 1px }`.

- [ ] **Step 2: Importar o CSS raiz**

Localizar onde o CSS global é importado (provavelmente `src/renderer/src/main.tsx` importa um `assets/*.css` ou `index.css`). Importar `./styles/tokens.css` e `./styles/base.css` (tokens antes de base). Manter o CSS existente do React Flow.

- [ ] **Step 3: UX fix — foco no centro do nó (CommandPalette)**

Em `CommandPalette.tsx`, o `run` de um item-nó usa `setCenter(n.position.x, n.position.y, …)` — centraliza no canto superior-esquerdo. Corrigir para o centro visual: `setCenter(n.position.x + (n.width ?? 0) / 2, n.position.y + (n.height ?? 0) / 2, { zoom: 1.2, duration: 400 })`.

- [ ] **Step 4: UX fix — label de nós-nota no palette**

Notas não têm `data.name`, então caem no literal `"note"`. Dar um label útil: para `type === 'note'`, `label = (data.content?.trim()?.slice(0, 24) || 'Nota')` — o início do conteúdo, ou "Nota". (Terminais/portais mantêm `data.name`.)

- [ ] **Step 5: UX fix — CommandPalette acessível (role + focus-trap + click-fora)**

No overlay: `role="dialog"` + `aria-modal="true"` no container; fechar ao clicar no backdrop (não no conteúdo); Tab/Shift+Tab ciclam dentro do modal (focus-trap simples: como só há o input + a lista, manter o foco no input é suficiente — garantir que Tab não sai para trás do overlay). Manter Esc fecha.

- [ ] **Step 6: UX fix — cores de status nos painéis**

Em `FloorsPanel.tsx` e `RoutinesPanel.tsx`, a linha de status hoje é sempre âmbar. Distinguir: sucesso → `var(--ok)`, erro → `var(--err)`. Guardar um `statusKind: 'ok' | 'err' | ''` junto do texto; renderizar a cor conforme. (Ex.: `land` ok → verde "aterrissado"; conflito/erro → vermelho.)

- [ ] **Step 7: UX fix — confirmação antes de remover um floor (evita perda de trabalho não-commitado)**

Em `FloorsPanel.tsx`, `remove` de um floor descarta o worktree (com `--force`). Adicionar uma confirmação **inline** (não `window.confirm`, que bloqueia): o botão "Remover" vira "Confirmar?" por ~3s ao primeiro clique (um `confirmingId` em estado); o segundo clique confirma. Evita remoção acidental de trabalho não-aterrissado.

- [ ] **Step 8: Testes + typecheck** — `npm test` (186/186 verdes — nada de comportamento mudou) + `npm run typecheck` limpo. (Se algum teste tocar o texto/estrutura alterada, ajustar o teste conforme o novo comportamento; a confirmação inline do floor pode precisar de um pequeno teste se houver teste do FloorsPanel — provavelmente não há.)

- [ ] **Step 9: Commit** — `git add -A && git commit -m "feat: design tokens + consolidacao de UX (foco no no, palette acessivel, cores de status, confirm de floor) (Fase 13)"`

## Notas de risco
- **Estética é subjetiva:** a direção (dark violeta premium) é uma escolha sensata para uma ferramenta de dev, mas o usuário deve validar/redirecionar no checkpoint — é o ponto onde a direção dele é insubstituível.
- **Zero regressão:** as mudanças são visuais; se algum teste assertar texto/estrutura alterada (ex.: o label de nota no palette, a cor de status), ajustar o teste ao novo comportamento, não o contrário.
- **Ícone PNG:** gerar um `.png`/`.icns` 1024² fiel a partir do SVG requer ferramenta de imagem; fica como passo de export do usuário (o SVG e o apontamento no config ficam prontos).
- **Inline styles → var():** migração incremental; se um componente ficar grande/confuso, extrair suas classes para um `.css` co-localizado é aceitável, mas não obrigatório.
